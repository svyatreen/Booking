# Полное описание проекта Selora (монорепозиторий)

Документ описывает назначение репозитория, технологии, структуру каталогов и файлов, а также то, что уже реализовано в коде. Сводка ориентирована на разработчика, который впервые открывает проект.

---

## 1. Назначение проекта

Это **веб-приложение для бронирования отелей**: пользователи могут просматривать отели и номера, оставлять отзывы, добавлять отели в избранное, оформлять бронирования; администраторы получают доступ к сводной панели (статистика, списки отелей, бронирований и пользователей). Репозиторий оформлен как **монорепозиторий на pnpm**: несколько пакетов (frontend, backend, общие библиотеки, вспомогательные артефакты) живут в одном дереве исходников с общим lockfile и общим каталогом версий зависимостей.

## 2. Технологический стек

### 2.1 Язык и инфраструктура сборки

- **TypeScript** (~5.9), общие строгие настройки в `tsconfig.base.json` (ES2022, `strictNullChecks`, и т.д.).
- **pnpm** как менеджер пакетов; в корневом `package.json` скрипт `preinstall` требует использовать именно pnpm.
- **Корневые скрипты**: `build` (typecheck + рекурсивная сборка пакетов), `typecheck` / `typecheck:libs`.

### 2.2 Бэкенд (`backend/`)

- **Node.js**, **ES modules** (`"type": "module"`).
- **Express 5** — HTTP API под префиксом `/api`.
- **PostgreSQL** через драйвер **`pg`** и **Drizzle ORM** — доступ к данным и миграции/синхронизация схемы (`drizzle-kit push`).
- **Аутентификация**: **JWT** (`jsonwebtoken`), пароли — **bcryptjs**.
- **Валидация входа**: схемы **Zod** (в т.ч. сгенерированные из OpenAPI и локальные обёртки в `backend/src/zod`).
- **Логирование**: **pino** + **pino-http**.
- **Сборка для продакшена**: **esbuild** (`build.mjs`), выход — `dist/index.mjs` с source maps.

### 2.3 Фронтенд (`frontend/`)

- **React 19** + **React DOM 19**.
- **Vite 7** — дев-сервер и сборка; плагины **@vitejs/plugin-react**, **@tailwindcss/vite**.
- **Tailwind CSS 4** — стили; типичный стек **shadcn/ui-совместимых** компонент на базе **Radix UI**, **class-variance-authority**, **tailwind-merge**, **clsx**.
- **Маршрутизация**: **wouter** (лёгкий роутер для SPA).
- **Данные с сервера**: **TanStack React Query**; HTTP-клиент и хуки **генерируются из OpenAPI** (Orval) в `frontend/src/api/generated/`.
- **Формы**: **react-hook-form**, **@hookform/resolvers**, **zod**.
- Дополнительно: **date-fns**, **react-day-picker**, **framer-motion**, **lucide-react**, **sonner**, **recharts**, **embla-carousel-react**, **next-themes** (тема оформления), и др.

### 2.4 Общие библиотеки (`lib/`)

| Пакет                         | Назначение                                                                                                                                                       |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `@workspace/db`               | Схема БД Drizzle для PostgreSQL (`lib/db/src/schema/*`), экспорт типов и zod-схем вставки через drizzle-zod; свои скрипты `push` / `push-force` для drizzle-kit. |
| `@workspace/api-spec`         | **OpenAPI 3.1** спецификация (`openapi.yaml`); скрипт **Orval** `codegen` генерирует клиент React Query и Zod-схемы.                                             |
| `@workspace/api-client-react` | Слой **сгенерированного** API-клиента (React Query + `custom-fetch.ts` для базового URL и токена).                                                               |
| `@workspace/api-zod`          | Сгенерированные **Zod**-типы/схемы для API (используются бэкендом через реэкспорт в `backend/src/zod`).                                                          |

### 2.5 Прочее

- **Prettier** в корне для форматирования.
- В `pnpm-workspace.yaml` заданы **workspace-пакеты**, **catalog** с закреплёнными версиями часто используемых зависимостей, политика **minimumReleaseAge** (минимальный «возраст» версии пакета на npm перед установкой — мера против supply-chain атак), а также **overrides** для уменьшения тянущихся платформенных бинарников (esbuild, lightningcss, rollup и т.д.).

## 3. Переменные окружения (ожидаемые приложениями)

- **`PORT`** — обязателен и для **frontend** (Vite), и для **backend** (слушающий порт). На деве фронт проксирует `/api` на `http://localhost:8080` (см. `frontend/vite.config.ts` — бэкенд ожидается на 8080 при такой настройке).
- **`DATABASE_URL`** — строка подключения к PostgreSQL; обязательна для `backend/src/db/index.ts` и для конфигов Drizzle.
- **`SESSION_SECRET`** — секрет для подписи JWT; при отсутствии в коде используется значение по умолчанию (для продакшена нужно задать явно).

## 4. Структура репозитория (верхний уровень)

```
Booking/
├── package.json              # Корень workspace, общие devDependencies, скрипты build/typecheck
├── pnpm-workspace.yaml       # Список пакетов, catalog, политики установки
├── pnpm-lock.yaml
├── tsconfig.json             # Project references на lib/db, lib/api-client-react, lib/api-zod
├── tsconfig.base.json        # Общие опции компилятора TypeScript
├── frontend/                 # Основное SPA-приложение
├── backend/                  # Основной Express API
├── lib/                      # Общие пакеты: db, api-spec, api-zod, api-client-react
├── artifacts/                # Дополнительные пакеты (копии/варианты API и UI)
├── scripts/                  # Вспомогательные скрипты (пример: hello.ts)
├── .gitignore
├── .npmrc                    # Настройки npm/pnpm для репозитория (если есть)
└── ОПИСАНИЕ-ПРОЕКТА.md       # Этот файл
```

В дереве также могут присутствовать служебные файлы среды разработки; они не являются частью бизнес-логики приложения.

## 5. Пакет `backend/` (детально)

### 5.1 Скрипты (`backend/package.json`)

- `dev` — сборка esbuild и запуск `dist/index.mjs` с source maps.
- `build` — только сборка.
- `start` — запуск собранного приложения.
- `db:push` / `db:push-force` — применение схемы Drizzle к БД через `drizzle.config.ts`.

### 5.2 Точка входа и приложение

- **`src/index.ts`** — читает `PORT`, запускает прослушивание; логирует ошибки через `lib/logger`.
- **`src/app.ts`** — создаёт Express-приложение: **pino-http**, **cors**, **json** / **urlencoded**, монтирует роутер на **`/api`**.

### 5.3 Маршруты (`src/routes/`)

Подключаются из `src/routes/index.ts`:

| Файл           | Назначение                                                                                          |
| -------------- | --------------------------------------------------------------------------------------------------- |
| `health.ts`    | Проверка живости (например `/healthz` в спецификации).                                              |
| `auth.ts`      | Регистрация и вход: хеширование пароля, выдача JWT и профиля пользователя без пароля.               |
| `users.ts`     | Профиль текущего пользователя, бронирования пользователя, админский список пользователей.           |
| `hotels.ts`    | Список отелей с фильтрами и сортировкой, CRUD для админа, похожие отели, статистика, комнаты отеля. |
| `rooms.ts`     | Операции с номерами (детали, список по отелю, создание/обновление/удаление для админа).             |
| `bookings.ts`  | Создание бронирований, список (в т.ч. админский), детали, отмена, оплата (по контракту API).        |
| `reviews.ts`   | Отзывы по отелю, создание, обновление, удаление.                                                    |
| `favorites.ts` | Список избранного, добавление/удаление по отелю.                                                    |

Валидация query/body во многих местах идёт через Zod-схемы из `../zod` (сгенерированные + согласованные с OpenAPI).

### 5.4 Авторизация (`src/middlewares/requireAuth.ts`, `src/lib/auth.ts`)

- **`requireAuth`** — заголовок `Authorization: Bearer <token>`, верификация JWT, заполнение `req.user`.
- **`requireAdmin`** — поверх `requireAuth` проверка роли `ADMIN`.
- **`auth.ts` (lib)** — `signToken` / `verifyToken`, `hashPassword` / `comparePassword`.

### 5.5 База данных (`src/db/`)

- **`index.ts`** — пул `pg`, экземпляр `drizzle` со схемой, реэкспорт схемы.
- **`schema/`** — таблицы **users**, **hotels**, **rooms**, **bookings**, **reviews**, **favorites** (дублируют по смыслу пакет `lib/db`; фактически одна и та же модель данных, путь к схеме указан в `backend/drizzle.config.ts`).

### 5.6 Zod (`src/zod/`)

- **`index.ts`** реэкспортирует сгенерированные API-схемы и типы из `generated/` (источник генерации — `@workspace/api-spec` / Orval).

### 5.7 Утилиты и одноразовые скрипты

В `src/` лежат **Node-скрипты** (расширения `.mjs` / `.ts`) для сидирования данных, обновления изображений и контента отелей/номеров — это вспомогательные инструменты наполнения/обслуживания БД, не часть runtime API.

### 5.8 Логирование

- **`src/lib/logger.ts`** — настройка pino для приложения.

## 6. Пакет `lib/db/` (`@workspace/db`)

- **`package.json`** — экспорты: корень и `./schema`.
- **`src/schema/users.ts`** — таблица `users`: email (unique), password, name, enum роли `USER` | `ADMIN`, `created_at`.
- **`src/schema/hotels.ts`** — отель: название, описание, город, адрес, рейтинг, звёздность, массивы удобств и изображений.
- **`src/schema/rooms.ts`** — номер: связь с отелем, тип (`single` | `double` | `deluxe` | `suite`), цена, гости, описание, изображения.
- **`src/schema/bookings.ts`** — бронь: пользователь, номер, даты заезда/выезда, итоговая цена, статус (`pending` | `confirmed` | `cancelled`).
- **`src/schema/reviews.ts`** — отзыв: пользователь, отель, оценка, текст.
- **`src/schema/favorites.ts`** — избранное: пользователь + отель (уникальная пара на уровне бизнес-логики обеспечивается API/БД).
- **`drizzle.config.ts`** — конфиг drizzle-kit для этого пакета.

---

## 7. Пакет `lib/api-spec/` (`@workspace/api-spec`)

- **`openapi.yaml`** — полное описание REST API: теги health, auth, users, hotels, rooms, bookings, reviews, favorites.
- Основные пути (префикс в спецификации — `/api` на сервере):
  - `/healthz`
  - `/auth/register`, `/auth/login`
  - `/users/me` (GET/PATCH), `/users/me/bookings`, `/admin/users`
  - `/hotels`, `/hotels/stats`, `/hotels/{id}`, `/hotels/{id}/similar`, `/hotels/{hotelId}/rooms`
  - `/rooms/{id}`
  - `/bookings`, `/bookings/{id}`, `/bookings/{id}/cancel`, `/bookings/{id}/pay`
  - `/hotels/{hotelId}/reviews`, `/reviews/{id}`
  - `/favorites`, `/favorites/{hotelId}`
- **`orval.config.ts`** — генерация в `lib/api-client-react/src/generated` (клиент React Query, mutator `customFetch`) и в `lib/api-zod/src/generated` (Zod).

## 8. Пакет `frontend/` (детально)

### 8.1 Вход и оболочка

- **`index.html`**, **`src/main.tsx`** — монтирование React.
- **`src/App.tsx`**:
  - **QueryClientProvider** (React Query, настройки retry/refetch).
  - **TooltipProvider**, **ThemeProvider**, **AuthProvider**.
  - **Wouter Router** с `base` из `import.meta.env.BASE_URL`.
  - Глобально: **Toaster**, **CookieConsent**.

### 8.2 Маршруты (страницы)

| Путь                                                                                                          | Компонент               | Примечание                     |
| ------------------------------------------------------------------------------------------------------------- | ----------------------- | ------------------------------ |
| `/`                                                                                                           | `home`                  | Главная                        |
| `/about`, `/press`, `/blog`, `/careers`                                                                       | информационные страницы |                                |
| `/help-center`, `/contact`                                                                                    | поддержка и контакты    |                                |
| `/cancellation-policy`, `/safety`, `/privacy-policy`, `/terms-of-service`, `/cookie-policy`, `/accessibility` | юридические/политики    |                                |
| `/login`, `/register`                                                                                         | вход и регистрация      |                                |
| `/hotels`                                                                                                     | каталог отелей          |                                |
| `/hotels/:id`                                                                                                 | карточка отеля          |                                |
| `/hotels/:hotelId/rooms/:roomId`                                                                              | деталь номера           |                                |
| `/favorites`                                                                                                  | избранное               | защищено `ProtectedRoute`      |
| `/profile`                                                                                                    | профиль                 | защищено                       |
| `/booking/:id`                                                                                                | деталь бронирования     | защищено                       |
| `/admin`                                                                                                      | админ-панель            | `ProtectedRoute` с `adminOnly` |
| fallback                                                                                                      | `not-found`             | 404                            |

### 8.3 Контексты

- **`AuthContext.tsx`** — токен в `localStorage`, синхронизация с `setAuthTokenGetter` в `custom-fetch`, запрос `getMe` для восстановления сессии, `login` / `logout`, флаги `isAuthenticated`, `isAdmin`, `isLoading`.
- **`ThemeContext.tsx`** — тема оформления (совместимо с `next-themes` в стеке UI).

### 8.4 API-слой

- **`src/api/index.ts`** — реэкспорт сгенерированного API и схем, настройка `setBaseUrl` / `setAuthTokenGetter`.
- **`src/api/custom-fetch.ts`** — обёртка над `fetch` для авторизации и базового URL.
- **`src/api/generated/`** — результат codegen (Orval): хуки `useX`, ключи запросов, типы ответов.

### 8.5 Компоненты

- **`components/layout/Layout.tsx`**, **`Navbar.tsx`**, **`Footer.tsx`** — каркас страниц.
- **`ProtectedRoute.tsx`** — редирект/блокировка для неавторизованных; вариант только для админа.
- **`CookieConsent.tsx`** — баннер согласия с cookies.
- **`components/ui/*`** — большой набор UI-примитивов (кнопки, диалоги, таблицы, формы, карточки, **hotel-card**, календарь, графики **chart** / recharts, и т.д.) в стиле дизайн-системы на Tailwind + Radix.

### 8.6 Хуки и утилиты

- **`hooks/`** — `use-mobile`, `use-debounce`, `use-toast`.
- **`lib/utils.ts`** — типичный `cn()` для классов.

### 8.7 Стили

- **`src/index.css`** — глобальные стили и интеграция с Tailwind v4.

### 8.8 Админ-страница (реализованная логика)

`admin.tsx` запрашивает списки отелей, всех бронирований и пользователей через сгенерированные хуки, считает выручку по подтверждённым бронированиям, показывает сводные карточки и таблицы (вкладки).

## 9. Каталог `artifacts/` (дополнительные пакеты)

Это **отдельные варианты или снимки** частей системы, которые собираются и типизируются вместе с workspace (см. корневой `typecheck`, фильтрующий `artifacts/**`).

### 9.1 `artifacts/api-server/`

- Отдельный **Express-сервер**, собираемый через свой `build.mjs`.
- Зависит от **`@workspace/db`** и **`@workspace/api-zod`** — то есть переиспользует общую схему БД и Zod-модели API.
- Содержит свои `src/routes/*`, `src/app.ts`, сиды (`seed-extended` и др.) — полезно как **изолированный сервисный пакет** или историческая ветка API.

### 9.2 `artifacts/hotel-booking/`

- **Vite + React** приложение, по зависимостям близкое к основному `frontend`, плюс зависимость на **`@workspace/api-client-react`**.
- Набор страниц включает, среди прочего, `login`, `admin` и т.д. — по сути **альтернативный или демонстрационный фронт** на том же API-контракте.

### 9.3 `artifacts/mockup-sandbox/`

- **Vite + React** песочница с полным набором UI-компонентов (mockup / прототип интерфейса).
- Есть кастомный плагин **`mockupPreviewPlugin.ts`** для сценариев превью при разработке макетов.

В подкаталогах артефактов встречаются служебные метаданные публикации пакетов как «артефактов» внешней платформы; на логику вашего приложения это не влияет.

## 11. Согласованность контракта API

Цепочка выглядит так:

1. **`lib/api-spec/openapi.yaml`** — единый контракт.
2. **Orval** генерирует типы и хуки для фронта и Zod для бэкенда.
3. **`backend`** импортирует Zod из своего `src/zod` (синхронизированного с генерацией).
4. **`frontend`** вызывает эндпоинты через сгенерированный клиент и прокси `/api` на бэкенд в режиме разработки.

При изменении API нужно обновлять OpenAPI и перезапускать codegen, затем подправлять реализацию роутов Express.

## 12. Что в итоге «реализовано» с точки зрения продукта

- **Модель данных** для отелей, номеров, пользователей с ролями, бронирований со статусами, отзывов и избранного.
- **REST API** по описанному выше набору путей (регистрация, вход JWT, профиль, отели с поиском/фильтрами, номера, брони, отмена/оплата по спецификации, отзывы, избранное, админские списки).
- **SPA** с публичными страницами, личным кабинетом, избранным, бронированиями и **админ-дашбордом**.
- **Инфраструктура** монорепозитория, строгая типизация, логирование на бэкенде, прокси для локальной разработки.
